规范

Naming Conventions: 信号使用全部小写字母；常数使用全部大写字母，大小写不敏感的编码；后缀用_n表示低电平启动，用_a表示异步（async），用_z表示三态，连接在一起的信号和端口使用完全相同的名字，千万不要使用HDL的保留字，保持组内、部门内、公司内的一致性。

File Header：必须包括以下内容：作者信息，修订历史，功能说明，可用参数，复位方法和时钟域，关键时序和异步接口，测试架构。

Ports：一个端口名占一行，并给出适当的注释；先写输入再写输出；顺序为时钟，复位，使能，其他控制，地址总线，数据总线。Port Mapping即端口映射方面看不懂。

Other Coding Guidelines: 适当的注释，小端字节序（使用[31:0]而不是[0:31]），操作数位宽要匹配，不要将信号赋上不关心的值，将所有寄存器全部reset【不知道RMM什么意思】。使用function来模型化组合逻辑电路（没看懂）。避免使用供应商限定的门电路。使用诸如Synopsis Getch或DesignWare这种不依赖具体技术的库。使用本地变量（local Variables）。

Clocks and Resets：避免同时使用时钟信号的两个边沿（占空比敏感，难以设计DFT）。绝不寄存时钟和复位信号（物理综合阶段做这个）。

Combinational/Sequential Blocks：对于组合电路模块（Combinational Block），使用阻塞赋值（blocking assignments），即=。最小化敏感列表（Sensitivity List）里面的信号数。赋值语句顺序需要与拓扑结构对应。对于顺序模块（疑似时序电路模块，Sequential Block），使用非阻塞赋值（non-blocking assignments），即<=。避免竞争问题（race problem）。组合电路模块和顺序模块要分离。

关于full_case和parallel_case属性

盘点哪些仿真与综合不匹配的RTL Coding Styles（一）

Coding for Synthesis：敏感列表的列举要准确且无冗余。避免使用full_case和parallel_case以避免pre-synthesis预综合和post-synthesis后综合的mismatch。避免在顶层模块使用胶水逻辑。另外在模块实例化调用特定port的时候不要在.()里面额外加逻辑，比如加一个MUX。

模块的命名：顶层模块命名为芯片名，次级模块命名为xx_core.v，例化名和模块名相同或加序号

信号的命名：信号用小写字母开头+下划线+数字，低电平有效的用_n或_b结尾，总线从高到低位命名如bus[31:0]注意reg [7:0] mema [15:0]表示16个8位的寄存器。信号名字不需要注明in还是out，只需要在注释里写，全局信号和接口信号应保持命名相同，模块例化采用信号名称连connection by name的形式，反正就是同名字点同名字。clk和rst等扇出大的信号列在信号列表最后。

同步触发器命名：用于同步异步信号时应加synch_

时钟：全局时钟叫clk，其他的时钟应下划线加频率。

文件：文件名与模块名相同

编码风格：initial begin和always begin不要单独给begin一行。时序单元用非阻塞。组合逻辑用阻塞。阻塞和非阻塞在一个block里面不能混用。敏感列表要完整。尽量不使用循环。

综合性：只使用一个主时钟。复位信号为异步rst_n。模块要保证输入和输出直接连接触发器，这是最好的。不在数据通路上的触发器都要有复位。数据通路上的触发器（可能因为时序紧张）复位根据流水线划分设置。同一个block要么都是带复位要么都不带复位。不使用casex和casez。case的第一条赋值和case在一行。用default。内部逻辑避免三态。不要在代码描述中加入specify去规定多周期路径。ifelse要避免生成锁存器。避免异步逻辑，带有反馈环的组合逻辑和自同步逻辑。输入和输出默认为wire。synopsys忽略initial综合。忽略所有的延时。

如何判断CPU是大端还是小端模式

小端：低字节存在低地址。大端：高字节存放在低地址。以16进制数字0x12345678为例，每个内存地址可以存储1byte也就是8位也就是2位16进制。

类型 小端 大端
地址 0x100 0x101 0x102 0x103 0x100 0x101 0x102 0x103
数据 0x78 0x56 0x34 0x12 0x12 0x34 0x56 0x78
地址 0x103 0x102 0x101 0x100 0x103 0x102 0x101 0x100
数据 0x12 0x34 0x56 0x78 0x78 0x56 0x34 0x12

小端模式 ：强制转换数据不需要调整字节内容，1、2、4字节的存储方式一样。
大端模式 ：符号位的判定固定为第一个字节，容易判断正负。

测试操作系统是大端还是小端的C++代码

测试大小端一般使用union的特性。union是一个联合体，所有变量公用一块内存，只是在不同的时候解释不同。其在内存中存储是按最长的那个变量所需要的位数来开辟内存的。在C里打0x12345678，由于小端，低位低地址高位高地址。再union一个char，这个char是存在低地址。因此小端机的chr存的是0x77。如果是char数组，那么就是[3 2 1 0]这样排，所以chra[0]存的是0x78。通常的x86+windows都是小端。

verilog入门经验（一） always块使用

任何在always块内被赋值的变量都必须是寄存器型（reg）。即<=或=左边的信号，必须是reg型，<=或=右边的信号可以是reg型也可以是wire型。

端口声明中被声明为input或inout型的端口，只能被定义为线网型（wire）；被声明为output型的端口，则可以被定义为线网型（wire）或者寄存器型（reg）。如果不定义，则默认为线网型（wire）。

always的敏感列表中可以同时包括多个电平敏感事件，也可以同时包括多个边沿敏感事件，但不能同时有电平和边沿敏感事件。另外，敏感列表中，同时包括一个信号的上升沿敏感事件和下降沿敏感事件也是不允许的，因为这两个事件可以合并为一个电平事件。

综上，负数补码不能单看整数部分最少多少位、小数部分最少多少位，加起来得到最终最少多少位。这一点在极限范围设计时非常重要。假设已知小数位数为fw，最佳解决方案是按bmwidth(fix(x*2^fw)来决定存储位数，必不会错。

因此针对小数的bmwidth函数可以用function binwidth = bmw_frac(fracnum,fracwidth)的形式，并加入decnum = fix(fracnum.*2.^fracwidth);【注意.*和.^】。

为什么Windows下1TB硬盘只有931GB？

硬件厂商说我这个盘是1TB，实际上就是1TB。但在windows下，显示虽然是TB，但是实际上是按TiB算的，也就是说1TB=0.9095TiB，而1TiB=1024GiB，所以Windows下就会显示931.328GiB（GB）了。

Windows显示5.44PB，实际上是5.44PiB，1PiB=1.1259PB，所以对于硬件厂商来说就是6.124896PB的硬盘。

硬件厂商和Windows所说的B都是8位，但是对K的定义有区别，因此导致后面的M G T都有区别。

需要进行两步换算：

首先1TB实际上是1TB，但是1GB实际上是1GiB。即问：1TB = 多少GiB？

1TB = 0.9095TiB

1TiB = 1024 GiB

1TB = 0.9095 * 1024 = 931.3GiB

或者

1TB = 1000GB

1GB = 0.9313GiB

1TB = 1000 * 0.9313 = 931.3GiB

因此在Windows系统下所有的KB，MB，GB全都指的是KiB MiB GiB

2^16 = 64 x 2^10 = 65536 2^32 = 4 x 2^10 x 2^10 x 2^10

2^64 = 16 x 2^60 2^10B = 1KiB 2^10 x 2^10B = 1MiB
