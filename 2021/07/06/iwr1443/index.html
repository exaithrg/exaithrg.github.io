System synchronization

SYNC_OUT：Low-frequency frame synchronization signal output. Can be used by slave chip in multichip cascading.

SYNC_IN： Low-frequency frame synchronization signal input.

5.9.2 Synchronized Frame Triggering

The IWR1443 device supports a hardware based mechanism to trigger radar frames. An external host can pulse the SYNC_IN signal to start radar frames. The typical time difference between the rising edge of the external pulse and the frame transmission on air (Tlag) is about 160 ns. There is also an additional programmable delay that the user can set to control the frame start time.

IWR1443 器件支持基于硬件的机制来触发雷达帧。 外部主机可以脉冲 SYNC_IN 信号以启动雷达帧。 外部脉冲上升沿与空中帧传输 (Tlag) 之间的典型时间差约为 160 ns。 还有一个额外的可编程延迟，用户可以设置它来控制帧开始时间。

$T_{active_frame}$是用户定义的；$T_{pulse}$最小是25ns，最大不能超过$T_{active_frame}$。

根据Yuji Sugimoto的讨论，他们所用的SYNC_IN信号是3.4V, 30ms period, 11us width。所以实际上我觉得用个100ns的应该也没问题。要求脉冲周期大于帧周期，即SYNC_IN pulse periodicity大于mmWare Studio的frame periodicity配置。如果单次触发的话应该不用考虑这个。

DB60M34A6435V1毫米波雷达核心板说明.pdf

此核心板是没有本公司提供的特制算法程序的，但是其内部的程序可以参考TI的官方例程（具有诸多的参考例程），不但可以进行例程演示，还可以通过CCS查看例程的源代码甚至在此基础上做进一步的修改开发，具体获得例程的方法请查阅TI官网。

这个核心板是一个最小的雷达系统，如果想要跑TI的例程，需要配套本公司的DBMMDEBUG调试板，这个调试板通过排针与核心板连接，两个板子组成一套基本上可以跑TI的所有例程，这个DBMMDEBUG调试板是XDS110仿真器，可以直接通过CCS进行程序的在线仿真，所以这里也不需要再购买TI的其他JTAG仿真器。

这个核心板为了减小体积以及降低成本，内部并没有将LVDS高速数据接口引出来，而引出来的串行接口如SPI、UART、CAN等，其传输速度较低不能满足实时ADC原始数据的输出，所以这个核心板不能实时获得ADC的原始数据，但是如果您对原始ADC数据的实时性没有要求，可以使用此核心板的串行接口慢慢打印原始ADC数据，这里举个使用UART打印原始ADC数据的例子，通过更改IWR6843内部程序，可以将ADC转换的数据存储到内部的L3 RAM中，这里假设ADC数据占用了全部的L3 RAM即768KB，UART波特率为921600bps，将所有ADC数据传输出来所花费的时间大致为768102410/921600≈8.54s（这里乘以10是因为串口发送有起始和停止位）。如果使用SPI进行传输的话，这里假设使用SPI的CLK为50MHz，打印所有ADC数据所花时间大致为76810248/50000000≈0.126s。所以只要对ADC数据的实时性没有要求的话，也是可以通过更改内部程序以及接口，实现原始ADC数据的输出。

客户可以自己根据供电的电源、通信接口自行设计底板（替代XDS110调试板），具体的接口说明后文会详细解释，用户根据核心板的尺寸、接口（使用UART、SPI或者是CAN等都可以）、系统电源（12V、5V）等设计好底板后，可以直接将核心板装上使用，注意一些接口以及电源一定符合核心板要求，如果可以的话本公司可以帮忙检查设计的底板。

为了避免核心板损坏，请严格注意以下几点：其一就是供电电压，一定不能使用超限的电源进行供电；其二是插拔核心板时一定要对系统断电；其三插拔核心板时要对齐排针和排母，不能错位插入，细心检查后再给系统上电（因为此核心板使用了防呆的排针，所以排针排母不对齐插入的现象应该不会再发生）；其四是测试过程中尽量不要用手触碰天线，可以的话请使用放电手环。

DB60M34A6435V1毫米波雷达核心板内部已经处理好了SOP0、SOP1、SOP2的默认状态，其中SOP0=1，SOP1=0，SOP2=0，其中SOP2通过排针引出，这里只能改变SOP2的状态，从而改变相应的运行模式。SOP2=0，雷达从Flash读取程序并正常运行；SOP2=1，UART向Flash烧程序。

数据包有32个帧（frame）；一帧有128个chirp；一个chirp采256个点；一个采样点16bit；一个接收天线分为I/Q两路采集；4个接收天线。

ADC是16位，4个接收天线（4路），adc data是16位int数以int16读入16777216字节，也就是8,388,608个数，故adcData刚开始是8388608x1；reshape后，每8个数据为一组，分了1048576组，也就是成了一个8x1048576的数据矩阵。readDCA1000()函数获取得到的数据矩阵为：4x1048576;其中4代表四个接收端；每一行的排列顺序为：先frame1的所有chirp，再frame2，类推到frame32。其中每一frame共有128个chirp，一个chirp是256个点；共有32frame。即：32x128x256=1048576个点。每个点实际上是2个16bit合成的，所以总点数是32framex128chirpx256pointsx16bitx2IQx4lanes/8bit=16777216Byte。1个frame就是512KB。

数据通过四个LVDS通道捕获，并以交错格式存储在二进制文件中。

out_of_box_demo是啥？

直接获取原始ADC数据，通过串口发送给客户。

SDK版本为mmwave_sdk_03_04_00_03

这个例程的GUI为mmWave_Demo_Visualizer

这里需要2个串口，分别为指令配置和数据传输串口（即User UART和Data Port），其中指令配置串口波特率为115200，数据传输串口波特率为921600，通过指令配置串口为毫米波雷达核心板配置启动指令，通过数据传输串口获得原始ADC数据。

通过指令配置串口所要发送的配置指令，如文件夹中“6843_2T4R_cfg”文件所示，如下图，因为数据传输串口的波特率为921600，传输数据速度有限，所以这里采取配置雷达只工作1个frame模式，下图中红色框中数值为1不允许改变，如果改为任何其他值都会导致雷达不能正常工作。关于其他配置参数，可按“6843_2T4R_cfg”文件中的参数配置，内部的参数可根据实际测量需要自行更改（参数更改要符合要求）。

正常情况下，下图所示的雷达配置参数在开机初始化配置一次即可，后面再次获取数据可以通过发送sensorStop和sensorStart 0指令重新获取新的数据。在雷达工作一帧并收到sensorStop指令前，其核心板的LED灯会点亮，在收到sensorStop指令后LED灯会熄灭，再收到sensorStart 0后LED又会点亮，以此循环，所以可以通过LED观察其工作状态。

发送的配置指令每一条后面必须有回车字符才能被雷达接收识别，配置每一条指令时间间隔要大于10ms。

注意，如果出现了配置参数后不能正常工作的情况，则需要重新启动雷达（给雷达重新上电），否则雷达不能恢复到正常工作流程。建议监视雷达回复配置指令或者是雷达回传数据，如果出现不能恢复的情况，请给雷达重新上电，可以通过雷达核心板上PMIC_EN管脚控制雷达重新上电。

不要用USB转串口一拖二线。问题可能有两个：

  1. USB有master和slave。一般来讲USB转串口一拖二的应用场景都是PC作为host，但是雷达板作为host的话可能不行。
  2. 哪怕雷达板可以当host，但是这种虚拟串口可能还存在芯片是否支持的问题。比如雷达板支持虚拟串口，PC也支持，所以可以用；但是BM3823可能不支持，就完蛋了。

而且网口的好处是速度快。

DCA1000 和 TSW 的操作应该没有区别，硬件触发和 SW 触发在 AWR 设备上，应该与使用的捕获设备无关。

在您开始提供 SYNC IN 脉冲后，您能否确认您正在毫米波工作室中执行“DCA1000 ARM”和“触发帧”吗？ 您提供的 SYNC IN 脉冲的周期是多少？帧配置中编程的帧周期是多少？

关键：嗨，维维克，通过在 DCA1000 上去掉 R120，我们已经能够解决这个问题。 该电阻将 SYNC_IN 脉冲连接到 DCA 的 FPGA 之一。我们看到我们输入的 SYNC_IN 信号通常具有 3.3V 的 pk 到 pk，但是当连接 DCA 时，它下降到 500mV。 将它连接到 DCA 会极大地衰减信号。 当电阻被拿走时，信号再次恢复到 3.3V pk 到 pk 并且触发器开始工作。

所以必须要拆掉R62才行。

IWR1443: iwr1443 boost external trigeering【

可以查到AR_SYNC_IN的接法。SYNC_IN是IWR1443的N10引脚，用于硬件触发。

根据IWR1443BOOST原理图.pdf，AR_SYNC_IN接了一个下拉电阻R196(10k)到SYNC_IN上。也就是说AR_SYNC_IN在不管的时候是低电平。

AR_SYNC_IN通过0欧的R62电阻接到了J1的16引脚上，J1是那个60pin的高速接口。

此外通过0欧的R165电阻接到了J5的9引脚上

】

关于硬件配置： 有两条路径可以提供硬件 Sync_in 信号：

1) 应连接此 R62 的 J1（60 针连接器）

2) 应连接此 R165 的 J5（20 针连接器）

上述 e2e 帖子的说明是：由于两个引脚都会到达毫米波传感器上的同一个球，因此您可以在一个引脚上施加信号，并可以在另一个引脚上进行监控。
